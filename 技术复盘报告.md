## 一、项目概述

本项目是一个基于火山引擎生态的**游戏 AI 助手**，核心形态为"纸片人智能体"，通过 RTC 语音交互实现实时对话、知识检索与视频推荐。系统从初始的单模型分类器，历经 8 个里程碑演进为 **FSM 驱动的分层异步并发智能体系统**。

### 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端 | Vanilla JS（模块化架构）、Live2D 桌宠 |
| 后端 | Node.js (v22/v24)、Express |
| AI/RTC | 火山引擎 Ark（LLM）、TTS V3（语音合成）、RTC（实时通话）、RAG 知识库 |
| 视频搜索 | 通用视频搜索（B站/抖音/Bing），强制返回可播放直链 |
| 脚本/运维 | PowerShell 5.x |

### 端口架构

| 服务 | 端口 | 启动方式 |
|------|------|----------|
| 前端静态服务 | 8081 | `static-server.js` |
| 后端 API 服务 | 8788 | `src/server.js` |

### 里程碑 6：FSM 状态机 + 分层异步 + 分级裁剪（架构大升级）

**时间：2026-05-12 深夜 ~ 2026-05-13 上午**

| 事件 | 详情 |
|------|------|
| 系统评估 | 对当前 Agent DEMO 进行 6 维度打分（**1.9/5**），明确从"工作流 Demo"到"自主智能体"的差距 |
| FSM 状态机 | 引入 `taskFsmService.js`，定义严谨状态流转：`CREATED → CONTEXT_LOADING → ROUTING → MAIN_REPLIED → BRANCH_QUEUED/EXEC → DONE` |
| task_id 任务隔离 | 所有事件必须绑定 `task_id`，严禁仅按 intent 或 turn_id 更新 UI 以防串线 |
| 并发池控制 | strategy/video 各限 **2 个并发**，引入排队机制 |
| Main_Agent Prompt 重写 | 新增 `emotional_reply`（情绪确认）、`understanding_reply`（理解摘要）、`branch_wait_reply`（等待语）分阶段反馈字段 |
| Output Trimmer | 新增 `outputTrimmerService.js`，定义 L0（语音）~ L3（调试）四级内容裁剪规则 |
| RAG 优先原则 | 确立所有 intent（含 smalltalk）均须基于 RAG 增强回复，Main_Agent 作为消费者直接使用 pre-fetched RAG 结果 |
| 按需生图 | 默认仅文字，显式要求才出图，降低延迟 |
| 预占位弹窗 | Main_Agent 返回前预创建任务和 Loading 弹窗，大幅降低感知延迟 |

**核心产出**：从"串行同步"到"FSM 驱动的分层异步并发"架构质变

**关键决策**：

- Main_Agent 需保持轻量化，仅负责路由与初步确认，保障极速首包回复
- 后端并发池控制比前端控制更可靠，能有效处理刷新或多入口触发的极端情况
- 将理解摘要（`understanding_reply`）与执行等待语（`branch_wait_reply`）分离播报，提供更自然的交互节奏
- Prompt 管输出质量，工程管响应速度，两者分工明确

**系统评估维度与得分**：

| 维度 | 得分 | 差距 |
|------|------|------|
| 基础链路 | 3/5 | 串行执行，无并行 |
| 任务规划 | 1/5 | 依赖 if/else 而非 LLM 自主规划 |
| 状态机 | 2/5 | 隐式状态，无显式 FSM |
| 工具集成 | 2/5 | 硬编码导入，无统一注册 |
| 记忆系统 | 2/5 | 仅 FIFO 裁剪，缺摘要压缩 |
| 反思闭环 | 1/5 | 无执行后自我修正 |

## 四、硬约束与工程约定

### 硬约束

| 约束 | 说明 |
|------|------|
| 任务隔离 | 必须使用 `task_id` 绑定所有事件，严禁仅按 intent 或 turn_id 更新 UI 以防串线 |
| FSM 状态机 | 任务必须遵循 `CREATED → CONTEXT_LOADING → ROUTING → MAIN_REPLIED → BRANCH_QUEUED/EXEC → DONE` 严谨流转 |
| 优先级调度 | 并发池支持 high/normal/low 优先级，high 可插队执行，同优先级 FIFO |
| 自动优先级 | 基于正则匹配（如"赶紧"、"待会"）自动判定任务优先级，无需 LLM 介入以降低延迟 |
| Strategy 重试 | 任务失败自动重试 2 次，间隔 1.5s，伴有 TTS 进度提示；重试耗尽则降级输出 |
| 视频直链 | 视频搜索必须返回可播放的直链（如 .mp4/.m3u8），支持 B站、抖音等多个平台 |

### 工程约定

| 约定 | 说明 |
|------|------|
| 思维链架构 | 采用"工程链路思维链"，通过多 Agent 串/并编排和 FSM 状态流转实现显式、可控的推理骨架 |
| Prompt 策略 | System Prompt 采用"结构化输出引导"而非 CoT，以降低 Token 消耗和延迟 |
| RAG 复用 | 上下文阶段完成 RAG 检索，Main_Agent 和子 Agent 必须共享该结果，严禁重复检索 |
| 异常监控 | SSE 事件流必须包含 `stream_end` 标记，用于前端判断后端是否正常结束或异常中断 |

## 五、经验教训汇总

| 类别 | 教训 |
|------|------|
| SSE 异常 | 异常断开时若不推送 error 事件，前端卡片会卡在 ROUTING 状态；需加强网络层错误捕获与日志记录 |
| 视频直链 | 仅靠页面链接无法实现流畅播放；Video_Agent 需通过多层解析（B站 BV 号/抖音 playAddr）提取直链 |
| TTS 过滤 | `IgnoreBracketText` 类型必须为 `Integer[]`，且需通过系统提示词约束 LLM 统一使用英文括号 |
| 环境变量 | 凭证缺失时服务静默降级而非报错，排查困难；修改后必须重启服务 |
| 签名问题 | AK/SK 不匹配时签名算法本身无误，需优先排查凭证配置 |
| ASR 去重 | 并发 ASR 触发导致请求冲突，需前端去重 + AgentModule 忙碌保护 |
| 状态映射 | UI 状态更新必须在业务结果确认后触发，不可提前设置成功状态 |
| RAG 去重 | 多个 Agent 服务可能重复调用 RAG，需在架构层面确立"一次检索、全局复用"原则 |

## 六、后续演进方向

基于 1.9/5 的系统评估，后续优化方向包括：

1. **反思闭环**：执行后自我评估与多轮修正机制
2. **FSM 显式化**：状态流转可视化与可调试
3. **工具注册中心**：统一工具注册、版本管理与异步处理
4. **记忆系统增强**：摘要压缩 + 长期用户画像
5. **分布式状态**：会话状态的分布式同步、并发锁及超时回收
6. **LLM 自主规划**：从 if/else 固定逻辑向 LLM 自主任务路径规划演进
